# Command Tool Metadata

- Status: `not-started`
- Created: `2026-03-14`
- Last updated: `2026-03-14`

- Execution mode: `mixed`
- Related docs: `docs/package-manifest-schema.md`, `scripts/install-mdt.js`, `commands/`, `cursor-template/commands/`

## Problem

The current command installation system has no way to declare which tools a shared
command supports. This produces two interconnected problems.

**Problem 1 â€” Duplicate file lists in package manifests.**
A command that should be installed for both Claude and Cursor must be listed twice:
once in `commands` (drives Claude install) and again in `tools.cursor.commands`
(drives Cursor install). Adding a new command to a package requires two edits per
package, and omitting the second silently skips Cursor.

**Problem 2 â€” Duplicate command files with no stated reason.**
`cursor-template/commands/` is a parallel directory of 17+ command files that mostly
mirror `commands/` but with tool-adapted wording. There is no machine-readable record
of whether a cursor-template file exists because it is genuinely different or because
it was copied and never diverged. Both cases look identical on disk.

The result is a system that is hard to maintain and hard to audit: it is not possible
to tell at a glance whether a command is available for Cursor, and it is not possible
to tell whether a cursor-template override is load-bearing.

## Goals

- [ ] Give each shared command a `<name>.meta.json` file that declares which tools it
  installs for (`"tools": ["claude", "cursor"]` etc.).
- [ ] Make `commands/` the single source of truth for all commands regardless of target
  tool. Move cursor-only commands from `cursor-template/commands/` to `commands/` with
  `"tools": ["cursor"]` metadata.
- [ ] Reduce package manifests to a single `commands` list. Remove `tools.cursor.commands`
  from all packages â€” the metadata makes it redundant.
- [ ] Keep `cursor-template/commands/` as a pure **override** directory: a file there
  overrides the shared version for Cursor only. An override file with no corresponding
  metadata entry or shared command is invalid and flagged by a new validator.
- [ ] Add a CI validator that enforces the new invariants.

## Non-goals

- [ ] Do not change the `skill.meta.json` schema or skill installation logic.
- [ ] Do not change how Codex commands are handled (Codex has no commands directory).
- [ ] Do not unify the content of cursor-template overrides with the shared versions â€”
  keeping genuine differences is expected and intentional.
- [ ] Do not change `tools.cursor.rules` or `tools.cursor.skills` in package manifests â€”
  this plan covers commands only.

## Proposed `command.meta.json` Schema

Each command file `commands/<name>.md` may have an adjacent `commands/<name>.meta.json`:

```json
{
  "tools": ["claude", "cursor"]
}
```

**Field reference:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `tools` | `string[]` | yes | Tools this command installs for. Valid values: `"claude"`, `"cursor"`. Codex has no commands directory and is never listed here. |

**Default behaviour when `.meta.json` is absent:** install for `["claude"]` only.
This preserves backwards compatibility â€” existing shared commands without metadata
continue to work as before.

**Cursor-template override:** if `cursor-template/commands/<name>.md` exists AND the
metadata includes `"cursor"`, the installer copies the override file (not the shared
file) to the Cursor commands directory. If the override file exists but the metadata
does NOT include `"cursor"`, the override is unused and the validator reports a warning.

## Installer Changes (`scripts/install-mdt.js`)

### Current flow (simplified)

```
installClaudeCommands:
  copy commands listed in packages[*].commands from commands/

installCursorToolCommands:
  copy commands listed in packages[*].tools.cursor.commands
  â†’ try cursor-template/commands/<name>.md first
  â†’ fall back to commands/<name>.md
```

### New flow

```
installCommandsForTool(tool, commandsDest, selectedPackages):
  for each command in packages[*].commands (deduplicated):
    meta = loadCommandMeta(commands/<name>.meta.json)  // defaults to { tools: ["claude"] }
    if tool not in meta.tools: skip
    src = cursor-template/commands/<name>.md  if tool === "cursor" AND override exists
          else commands/<name>.md
    copy src â†’ commandsDest/<name>.md
```

`installClaudeCommands` and `installCursorToolCommands` are replaced by a single
`installCommandsForTool(tool, ...)` call. The `getToolManifestSelections` call for
`tools.cursor.commands` is removed from the Cursor path.

### New helper: `loadCommandMeta(filePath)`

```js
function loadCommandMeta(metaPath) {
  if (!fs.existsSync(metaPath)) {
    return { tools: ['claude'] };
  }
  const raw = JSON.parse(fs.readFileSync(metaPath, 'utf8'));
  return { tools: Array.isArray(raw.tools) ? raw.tools : ['claude'] };
}
```

## Package Manifest Changes

`tools.cursor.commands` is removed from all package manifests. The single `commands`
list drives installation for all tools; the per-command metadata controls which tools
each command reaches.

Before:
```json
{
  "commands": ["plan.md", "verify.md", "commit.md"],
  "tools": {
    "cursor": {
      "commands": ["plan.md", "verify.md", "commit.md", "security.md", "build-fix.md"]
    }
  }
}
```

After:
```json
{
  "commands": ["plan.md", "verify.md", "commit.md", "security.md", "build-fix.md"],
  "tools": {
    "cursor": {
      "rules": ["..."]
    }
  }
}
```

Notes:
- Commands that were only in `tools.cursor.commands` (e.g. `security.md`) move to the
  shared `commands` list but carry `"tools": ["cursor"]` metadata so they do not appear
  in Claude installs.
- Commands shared between tools carry `"tools": ["claude", "cursor"]` metadata.

## Migration Path

### Phase 1 â€” Metadata files for all existing shared commands

For every file in `commands/`:

1. Determine current install targets by cross-referencing:
   - Is the file listed in `tools.cursor.commands` in any package? â†’ add `"cursor"`.
   - Is the file in `commands/` (always)? â†’ add `"claude"`.
2. Write `commands/<name>.meta.json` with the determined `tools` array.

Files currently only in `cursor-template/commands/` (no shared equivalent):
- `install-rules.md`
- `security.md`

These move to `commands/` with `"tools": ["cursor"]` and are added to relevant package
`commands` lists.

### Phase 2 â€” Installer refactor

1. Add `loadCommandMeta` helper.
2. Rewrite `installClaudeCommands` and `installCursorToolCommands` as a single
   `installCommandsForTool(tool, commandsDest, selectedPackages)`.
3. Remove `getToolManifestSelections` call for `tools.cursor.commands` in the Cursor
   install path.
4. Retain the baseline shared command install (existing `installBaselineSharedCommands`)
   â€” it still works as-is since baseline commands will have metadata declaring their
   targets.

### Phase 3 â€” Package manifest cleanup

Remove `tools.cursor.commands` from all 7 language package manifests. Move any
cursor-only commands (`security.md`, `build-fix.md`, `refactor-clean.md`, etc.) from
`tools.cursor.commands` into the shared `commands` list.

### Phase 4 â€” New CI validator

Add `scripts/ci/validate-command-metadata.js`:

- For every `.md` in `commands/`, assert a `.meta.json` exists beside it.
- Assert each `.meta.json` is valid JSON with a non-empty `tools` array containing
  only known values (`"claude"`, `"cursor"`).
- For every `.md` in `cursor-template/commands/`, assert a corresponding `.meta.json`
  in `commands/` includes `"cursor"` in its tools list (override without metadata is
  invalid).
- Report all violations; exit 1 if any found.

Add the validator to `SCRIPT_STEPS` in `scripts/ci/run-test-pipeline.js`.

### Phase 5 â€” Test updates

- Update `tests/scripts/install-mdt-unit.test.js` to reflect removal of
  `tools.cursor.commands` from package manifests.
- Add unit tests for `loadCommandMeta` (missing file, valid file, invalid tools array).
- Add tests for the new `installCommandsForTool` covering: claude-only command, cursor-
  only command, dual-tool command with and without cursor-template override.

## Open Questions

- [ ] Should `tools.cursor.commands` entries be tolerated as an override mechanism
  during a deprecation window, or removed hard in Phase 3?
  **Tentative decision:** remove hard â€” keeping both paths creates the exact confusion
  this plan is designed to eliminate.

- [ ] Should the default (no `.meta.json`) be `["claude"]` or `["claude", "cursor"]`?
  **Tentative decision:** `["claude"]` only â€” explicit opt-in for Cursor prevents
  accidentally deploying commands that haven't been reviewed for Cursor compatibility.

- [ ] Do we want a `claude-template/commands/` override mechanism symmetrical to
  `cursor-template/commands/`? Currently it holds a few files already.
  **Tentative decision:** defer â€” out of scope for this plan, revisit when
  `claude-template/` work (see `20260313.fix-claude-plugin-and-template.md`) lands.
