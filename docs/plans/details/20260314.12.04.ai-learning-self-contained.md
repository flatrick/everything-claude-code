## Goal

- [ ] Use the dependency/support contract to make `ai-learning` a single canonical skill source with a shared runtime core, thin hook adapters, stable internal paths, and identical core behavior across repo, temp/dev copies, and installed homes.

## Current Failures

- [ ] `mdt learning analyze` can time out silently because the outer CLI kills a longer-running analyze flow.
- [ ] The current analysis prompt can hang tool-enabled CLIs by asking the model to `Read` files interactively.
- [ ] Wrapper scripts still rely on fragile multi-candidate path resolution chains.
- [ ] `ai-learning` currently drifts across shared skill files, Codex overlay files, and runtime code outside the skill tree.
- [ ] Reusable logic is not yet cleanly separated from hook-only adapters.

## Non-goals

- [ ] Do not define the generic dependency/support schema in this plan.
- [ ] Do not create a second runtime tree for Codex or any other tool.
- [ ] Do not start the compose redesign here.
- [ ] Do not update current-truth docs outside planning until implementation lands.

## Decisions

- [x] Keep one canonical repo source for `ai-learning` under `skills/ai-learning/`.
- [x] `ai-learning` consumes the dependency/support contract from the prerequisite plan rather than defining its own local metadata model.
- [x] Shared runtime logic must live in a skill-owned runtime core. Hook files are thin adapters only.
- [x] Any logic needed by non-hook flows must move out of `hooks/` into the shared runtime core.
- [x] Per-tool add-ons may shape only metadata, docs, and config in template roots. They must not duplicate runtime code or create a second skill implementation.
- [x] `mdt.js` keeps outer skill-root discovery for installed layouts. Only internal skill imports need to become stable and relative.
- [x] Analyzer payload transport must use stdin or a bounded temp-file flow. Do not depend on model Read-tool calls and do not rely on giant argv prompts.
- [x] The optional Codex observer remains a separate add-on package resolved via the dependency/support model.
- [x] Tool-varying defaults such as observer settings should come from config add-ons/fragments, not copied skill trees.
- [x] Reuse shared project/runtime contracts where possible; do not introduce `ai-learning`-only dependency or install conventions here.

## Risks

| Risk | Severity | Mitigation |
|------|----------|-----------|
| `mdt.js` dynamic `require()` makes LSP blind to callers | High | Cross-check with `rg` after Serena symbol searches |
| `codex-template/skills/ai-learning/` has Codex-specific config (`openai.yaml`) not present in `skills/ai-learning/` | High | Port `openai.yaml` into `skills/ai-learning/` as a per-tool config fragment before deleting the duplicate tree |
| `scripts/codex-observer.js` will break if `continuous-learning/` moves without updating its require path | High | Update all callers in the same commit as the move |
| Analyzer stdin/temp-file approach may behave differently across Claude, Cursor, Codex environments | Medium | Test explicitly in all three before closing workstream |
| `runInstallForTarget()` in `install-mdt.js` copies whatever files land in the skill tree — moving paths could change install output | Medium | Verify closure snapshot passes after Phase 1 and Phase 2 |

## Implementation Phases

The workstreams below map to six ordered phases. Later phases depend on earlier ones;
do not reorder.

```
Phase 1 (shared runtime core)
  ↓
Phase 2 (retire codex-template duplicate)
  ↓
Phase 3 (stable entrypoints)
  ↓
Phase 4 (analyzer fixes)
  ↓
Phase 5 (metadata/installer integration)
  ↓
Phase 6 (tests + docs)
```

### Phase 1 — Move runtime core into skill tree

**Objective:** `scripts/lib/continuous-learning/` → `skills/ai-learning/lib/continuous-learning/`

Steps:
1. `git mv scripts/lib/continuous-learning/ skills/ai-learning/lib/continuous-learning/`
2. Update `mdt.js` require at line ~10:
   - Old: `require('./scripts/lib/continuous-learning/...')`
   - New: `require('./skills/ai-learning/lib/continuous-learning/...')`
   - Use `find_referencing_symbols` + `rg` cross-check to catch all callers.
3. Update `scripts/codex-observer.js` — it requires learning scripts directly. Point to new location.
4. Verify no remaining references to the old path: `rg "scripts/lib/continuous-learning"`.
5. Run `npm test` — must stay green.

### Phase 2 — Retire codex-template duplicate

**Objective:** `codex-template/skills/ai-learning/` removed; its unique config ported into `skills/ai-learning/`.

Steps:
1. Audit what is unique in `codex-template/skills/ai-learning/` vs `skills/ai-learning/`:
   - `openai.yaml` or similar Codex-specific config → port into `skills/ai-learning/` as a per-tool config fragment.
   - Any runtime JS files → confirm they are exact duplicates or differ only in path references.
2. Copy unique config fragments into `skills/ai-learning/` under an appropriate subdir.
3. `git rm -r codex-template/skills/ai-learning/`.
4. Verify `npm test` still passes and the resolver closure snapshot is unchanged (or update it).

### Phase 3 — Stable entrypoints and path rules

**Objective:** All internal `ai-learning` imports are stable relative paths; no multi-candidate chains.

Steps:
1. `rg "require\(" skills/ai-learning/` — list every `require()` call inside the skill tree.
2. For each call that uses an absolute or multi-candidate path, replace with a relative import from the new `lib/` location.
3. Verify hook files (`hooks/scripts/`, `hooks/cursor/`) only call into `skills/ai-learning/lib/` — no runtime logic inline.
4. Add a smoke assertion: load the skill's main entrypoint from a temp copy to confirm relative paths resolve from an installed home.
5. `npm test` green.

### Phase 4 — Analyzer and runtime behavior fixes

**Objective:** `mdt learning analyze` works reliably without timeout or Read-tool hang.

Steps:
1. **Timeout fix:** In `mdt.js` (or the analyze entrypoint), add an explicit `--timeout` guard that gives the analyze flow enough headroom (≥ 60 s). The outer CLI's default kill should not fire before the LLM call completes.
2. **Payload transport:** Replace any prompt that asks the model to `Read` files interactively:
   - Capture the relevant file content server-side.
   - Deliver it via `stdin` pipe or a bounded temp file passed as an argument.
   - The model receives content inline; no approval hang, no oversized `argv`.
3. **Hook vs explicit capture:** Confirm hook-enabled tools (Claude, Cursor) go through hook capture; hook-free tools (plain shell, Codex) go through explicit capture. Both must use the same shared runtime functions post-Phase 1.
4. `npm test` green; manual smoke on at least two tools.

### Phase 5 — Metadata and installer integration

**Objective:** `deps.json` sidecars accurately reflect the post-Phase-1/2 layout; closure snapshot updated if needed.

Steps:
1. Re-run `node scripts/ci/validate-dependency-sidecars.js` — must pass.
2. Re-run `node scripts/ci/validate-resolver-closure.js` — if the move changed install output, regenerate snapshot with `node scripts/ci/validate-resolver-closure.js --update-snapshot`.
3. Confirm `packages/ai-learning/deps.json` still declares `skill: ai-learning` correctly.
4. Confirm `packages/continuous-learning-observer/deps.json` still extends `ai-learning`.
5. `npm test` green.

### Phase 6 — Tests and docs

**Objective:** Tests verify canonical source and target-shaped outputs; docs reflect new layout.

Steps:
1. Add/update tests that assert:
   - The runtime core exists under `skills/ai-learning/lib/`.
   - No runtime JS remains under `codex-template/skills/ai-learning/`.
   - Hook files contain no load-bearing logic (are thin adapters only).
2. Update `skills/ai-learning/SKILL.md` — revise any path references to the old `scripts/lib/continuous-learning/` location.
3. Update `docs/plans/active.md` status to track completion.
4. After all verification items pass, follow the plan completion gate (ask operator A/B).
